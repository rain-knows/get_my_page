## 1. 为什么需要多介质存储（混合持久化）？
在目前设计的这套现代化架构中，我们彻底抛弃了“什么都往 MySQL 塞”的老旧做法，实现了**混合持久化 (Polyglot Persistence)**。

## 2. 数据资产分布矩阵

| 数据/资产类型 | 主要存储层 | 补充/关联存储层 | 更新/同步机制 | 故障降级策略 |
|--------------|-----------|---------------|--------------|-------------|
| **用户基础信息** (账号/密码) | `MySQL` (user 表) | 无 | 直接写入 | - |
| **文章长文本** (MDX内容) | `MySQL` (post 表) | `Caffeine` / `Redis` | MySQL 落盘后写入 L2 (Redis) | 若 Redis 挂掉，查 MySQL |
| **图片/附件资源** (头像/插图) | `MinIO` (文件对象) | `MySQL` (存最终 URL) | 上传 MinIO 成功后保存 URL | 静态资源不降级 |
| **全文搜索高亮索引** | `Meilisearch` | `MySQL` (基准数据源) | 发布文章时触发**异步增量推拉** | 若引擎挂掉，切回 MySQL 模糊查询 |
| **阅读量 / 点赞数** | `Redis` (原子 Hash/ZSET) | `MySQL` (post 表定期防丢) | 点击立即写 Redis，通过定时器 `CRON` 定期回写表 | 后台回写防止前端卡顿 |
| **鉴权 Tokens / 验证码** | `Redis` | 无 | 无状态存储，附带强设定 TTL 过期时间 | 必须强依赖 Redis |
| **热门文章排行版** | `Redis` (ZSET) | `MySQL` (数据支撑) | 根据浏览点赞算法定期全量刷新 | 切回按照最新时间兜底 |

## 3. ID 绑定与孤儿防范策略

当一份带有上下文的复合数据（如：一篇含有封面的热点博文）在被写入时，它将跨越所有的存储层：

### 3.1 最佳保存链路（单次全流程写入）
1. [前端] 向 `MinIO` 申请直传/服务端中转上传该博文图片，获得持久化的 `URL`。
2. [后端] 接受到包含图片 `URL` 和文字内容的载体，落入 `MySQL` 获得分配的绝对标识 `ID`。
3. [后端] 向 `Redis` 声明 ID 的存在（初始化点击量计数器为 0）。
4. [后端] 异步抛出一个事件或消息。
5. [异步消费者] 捕获到这篇含有 `ID` 的文章流，对齐推送给 `Meilisearch` 建立索引。

### 3.2 脏数据（孤儿资产）容错
*   **MySQL 删除了，但照片留在了 MinIO 占空间怎么办？**：
    由于云硬盘便宜且照片不敏感，默认逻辑删除 MySQL 记录即可，依靠 MinIO 自身的 LifeCycle 定期清理无引用 Bucket，无需在后端写死同步删除逻辑避免强耦合。
*   **Meilisearch 和 MySQL 内容不一致？**：
    后端提供一个专门的 Admin 接口，支持“一键全量重建索引”，利用分页向 MySQL 请求数据并覆盖注入 Meilisearch。

## 4. 深度聚焦：图片与多媒体资产流转

图片（封面、插图、用户头像）通常是博客系统占据存储空间和带宽消耗最大的部分。我们对此制定了如下防阻塞的专项策略：

### 4.1 核心载体与压缩分发机制
*   **存储引擎**：只存在 `MinIO` 中的 `blog-assets` 等隔离 Bucket（桶）内。绝不将 Base64 或二进制存放进任何数据库。
*   **加速与裁剪处理分发层 (Next.js Image)**：这是决定博客体感的重点！在前端消费图片时，我们绝不让浏览器直接生硬地去 `MinIO` 拉取数MB的原图。所有的图片渲染均经过 `Next.js` 强大的原生 `<Image />` 组件（`next/image`）包裹拦截。
    *   **效果**：当一张 5MB 的原生 4K PNG 封面被调用时，`Next.js` 底层节点会将其在首屏渲染期间压缩并剪裁成 WebP，再发送一张约 80KB 的图给浏览器，省去 95% 流量！

### 4.2 直拉与直传链路 (Presigned URL)
由于后端的 Tomcat 服务器带宽极度宝贵，我们不能让所有的千兆大图“通过后台再转存进 MinIO”。
**最佳实践（直塞法）**：
1. **签发阶段**：前端向 Spring Boot 索要凭证 -> “我想传一张名为 `header.png` 的图”。Spring Boot 直接向本地 MinIO 验证后，颁发一张带有超时的“免密放行直传链接 (Presigned Put URL)”。
2. **脱离后端**：前端直接将二进制文件推向 MinIO 被加密保护的网关，完全不经过 Spring Boot 这个中转站！
3. **反哺数据**：前端将拿到的最终公开访问短 URL，通过表单存入表内。

### 4.3 存储目录区隔规范
对于上传进 MinIO 的对象资源，强制规范其前缀结构：
*   **头像类**：`/avatars/{user_id}/{timestamp}.jpg`
*   **文章封面**：`/covers/{YYYY}/{MM}/post_{post_id}_{hash}.jpg`
*   **富文本内图**：`/articles/{YYYY}/{MM}/{post_id}/{uuid}.png`

---

> **结语**: 该矩阵定义了我们业务数据的去向，所有的底层 Repository/Mapper 定义需以此为界限，绝对禁止越权存储。
