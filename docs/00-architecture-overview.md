## 1. 当前阶段定位

本项目当前阶段是 **学习驱动的 MVP 构建期**，目标不是一次性做完整博客平台，而是完成一条可验收、可复盘、可扩展的核心业务闭环。

### 1.1 阶段目标

| 目标维度 | 阶段目标 |
|---|---|
| 功能闭环 | 打通认证、文章、评论、搜索、上传、预发布主链路 |
| 学习闭环 | 每个能力点都映射到可执行 Step 与验收证据 |
| 工程闭环 | 需求（docs）-> 执行（learn）-> 验收（命令）-> 复盘（记录） |

### 1.2 本阶段成功标准

- 关键 API 可按契约返回统一响应与错误码。
- 每条 P0 需求至少覆盖“成功 + 失败”两类验收路径。
- 文档体系无失效链接，且可追踪到具体实现与证据目录。

## 3. 关键业务链路（MVP）

### 3.1 认证链路

- 注册/登录 -> 颁发 access + refresh token。
- refresh token 刷新 access token。
- 登出触发 token 撤销策略（黑名单或等效方案）。

### 3.2 内容链路

- 公开读：文章列表分页 + 文章详情（slug）。
- 管理写：文章 CRUD + 分类/标签关联。

### 3.3 互动与检索链路

- 评论：游客/登录双模式写入 + 树形读取。
- 搜索：关键词查询 + 高亮返回。

### 3.4 资源与发布链路

- 上传：文件写入 MinIO 并返回可访问 URL。
- 预发布：构建、测试、编排检查与回归记录。

## 4. 分层原则（实施层面）

### 4.1 前端

- `app`：只负责路由与页面拼装。
- `features/*`：业务逻辑、业务组件、业务状态优先落位。
- `components/ui`：仅基础 UI 原子组件，不承载业务语义。

### 4.2 后端

- `controller`：入站协议与参数接收。
- `service`：业务规则与跨组件编排。
- `mapper`：数据访问。
- `infrastructure`：缓存/搜索/存储等外部系统接入。

详细约束请以 [架构契约](./architecture-contract.md) 为准。
